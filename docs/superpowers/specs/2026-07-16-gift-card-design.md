# CoffeeBar 购物卡充值与混合支付设计

## 目标

为已登录用户提供一张 CoffeeBar 购物卡。用户可以选择 ¥100、¥200、¥300 或 ¥500 进行模拟充值；结算时可手动选择使用购物卡。余额足够时由购物卡全额支付，余额不足时自动用尽购物卡余额，并以现有模拟支付补足差额。

本功能必须持久化余额和每次资金变动，避免重复充值、重复扣款或并发支付导致负余额。

## 产品规则

- 充值和订单差额支付均为模拟支付，不接入真实支付渠道。
- 只允许固定充值金额：¥100、¥200、¥300、¥500；充值金额按面值到账，无赠送金额。
- 充值入口位于“我的”页面，并提供独立购物卡页面展示余额、充值选项和最近流水。
- 只有已登录用户可以充值或使用购物卡。
- 结算时由用户手动勾选“使用购物卡”；未勾选时，订单全部走现有模拟支付。
- 勾选后，购物卡最多抵扣订单应付金额：
  - 余额大于等于订单金额：购物卡全额支付，模拟付费金额为 0。
  - 余额小于订单金额：购物卡余额全部抵扣，剩余金额走模拟支付。
  - 余额为 0：不能启用购物卡支付。
- 充值不计入会员累计消费；完成订单后仍按订单总金额累计会员消费，不因支付方式拆分而改变。
- 本期不支持退款、撤销充值、购物卡转赠、失效日期、优惠赠送或管理员调账。
- 所有金额继续使用整数“分”存储和计算。

## 数据模型

### GiftCardAccount

每位用户最多一个购物卡账户：

- `id`: 主键。
- `userId`: 用户 ID，唯一约束。
- `balance`: 当前可用余额，整数分，默认 0。
- `createdAt`、`updatedAt`: 创建与更新时间。

`User` 增加可选的一对一 `giftCardAccount` 关系。

### GiftCardTransaction

购物卡资金流水：

- `id`: 主键。
- `accountId`: 所属购物卡账户。
- `type`: `RECHARGE` 或 `PURCHASE`。
- `amount`: 有符号整数分；充值为正数，订单抵扣为负数。
- `reference`: 幂等引用，唯一约束。充值使用客户端充值令牌，消费使用订单结算令牌派生引用。
- `orderId`: 消费流水关联的订单，可空；同一订单最多一条购物卡消费流水。
- `createdAt`: 创建时间。

流水不可编辑或删除。当前余额保存在账户表以支持快速读取和原子扣减，流水用于审计资金来源与去向。

### Payment

保留一笔订单对应一条 `Payment` 的现有边界，并扩展支付拆分字段：

- `amount`: 订单总实付金额，保持现有语义。
- `giftCardAmount`: 购物卡支付金额，默认 0。
- `externalAmount`: 现有模拟支付金额，默认等于历史记录的 `amount`。
- `providerRef`: 模拟支付引用；购物卡全额支付时允许为空。

迁移时为历史 `Payment` 设置 `giftCardAmount = 0`、`externalAmount = amount`，保证后台收入统计和历史订单不变。

## 服务端边界

### 查询购物卡

新增只读查询，返回当前登录用户的余额和最近流水。尚未创建账户时按余额 0 返回；首次充值或消费时再创建账户。

### 模拟充值

新增服务端 action，接收 `amount` 和唯一 `token`：

1. 校验登录状态、固定金额白名单和令牌格式。
2. 在数据库事务中按 `userId` 获取或创建购物卡账户。
3. 若 `token` 对应的充值流水已经存在，返回原结果，避免双击或网络重试重复到账。
4. 原子增加账户余额并写入一条正数 `RECHARGE` 流水。
5. 返回新余额并刷新购物卡与个人中心页面。

客户端提交的金额不能直接绕过白名单；只允许 `10000`、`20000`、`30000`、`50000` 分。

### 订单支付

`CheckoutInput` 新增布尔值 `useGiftCard`。服务端仍以数据库中的商品和规格重新计算订单总额，绝不接受客户端计算的支付拆分。

在现有库存扣减和订单创建事务内：

1. 如果未启用购物卡，`giftCardAmount = 0`，`externalAmount = totalAmount`。
2. 如果启用，读取或创建用户购物卡账户，计算计划抵扣额 `min(balance, totalAmount)`。
3. 通过带 `balance >= plannedAmount` 条件的原子更新扣减余额，禁止余额变为负数。
4. 如果并发操作导致条件更新失败，整个事务回滚并提示“购物卡余额已变化，请重试支付”。
5. 创建订单、订单明细、支付拆分，以及一条负数 `PURCHASE` 购物卡流水。
6. 购物卡全额支付时不生成模拟渠道引用；混合或纯模拟支付时生成现有模拟渠道引用。

现有 `checkoutToken` 唯一约束继续保证订单幂等。重复请求返回已经创建的订单及其原始支付拆分，不再次扣减余额或库存。

## 页面与交互

### “我的”页面

在会员概览下增加购物卡卡片，展示当前余额和“去充值”入口。它不混入“累计消费”统计卡，避免把储值误解为消费。

### 购物卡页面

新增 `/profile/gift-card`：

- 顶部展示当前余额。
- 展示 ¥100、¥200、¥300、¥500 四个固定金额选项。
- 用户选择金额后，通过确认弹窗完成模拟充值。
- 成功后更新余额并展示成功提示。
- 下方展示最近流水，区分“充值”和“订单消费”，显示正负金额和时间。
- 未登录时跳转登录页，并携带返回地址。

### 结算页面

支付摘要增加“购物卡”区域：

- 展示可用余额。
- 余额大于 0 时可手动勾选“使用购物卡”；默认不勾选。
- 勾选后实时展示购物卡抵扣与模拟付费两行金额。
- 余额不足时明确提示“购物卡抵扣 ¥X，剩余 ¥Y 使用模拟支付”。
- 最终确认弹窗展示相同拆分；服务端返回结果后以服务端拆分为准。

### 支付成功页

展示订单总额，并在使用购物卡时补充购物卡支付金额与模拟付费金额，便于用户核对混合支付结果。

## 演示模式

“模拟支付”表示支付渠道不产生真实扣款，但购物卡账户与流水仍要求数据库持久化。未配置数据库的静态演示模式只展示购物卡入口和固定面额，不伪造可持续余额；提交充值或使用购物卡时明确提示需要配置数据库。现有纯模拟订单能力保持不变。

## 错误处理

- 非固定充值金额：拒绝并提示可选面额。
- 未登录：引导登录后返回原页面。
- 重复充值或结算令牌：返回第一次成功结果，不重复记账。
- 并发余额变化：回滚订单、库存和扣款，提示用户重新确认。
- 商品、规格或库存变化：沿用现有结算错误，购物卡不发生扣款。
- 任一数据库写入失败：整个事务回滚，不允许订单、余额与流水出现部分成功。

## 测试与验收

### 单元测试

- 固定充值金额白名单接受 100/200/300/500 元并拒绝其他金额。
- 支付拆分在不使用购物卡、余额充足、余额不足、零余额四种情况下正确。
- 金额全部使用整数分，无浮点计算。

### 服务端行为测试

- 重复充值令牌只入账一次。
- 重复结算令牌只扣减一次购物卡余额和库存。
- 并发扣减不能产生负余额；失败事务不创建订单或流水。
- 纯购物卡、纯模拟支付、混合支付均正确写入 `Payment` 拆分。

### 页面验收

- “我的”页面能进入购物卡页面并看到余额。
- 四个固定面额均可模拟充值，充值后余额和流水立即更新。
- 结算默认不使用购物卡；用户勾选后金额拆分即时更新。
- 余额充足时模拟付费为 0；余额不足时自动显示并支付差额。
- 支付成功页展示与数据库一致的支付拆分。
- 中文与英文界面均有完整文案，移动端与桌面端布局可用。

## 成功标准

用户能够完成“固定面额模拟充值 → 查看余额和流水 → 手动选择购物卡结算 → 余额不足时混合支付 → 查看支付拆分”的完整闭环；所有资金变化可追溯，且重复请求、并发请求或订单失败不会导致重复入账、重复扣款或负余额。
